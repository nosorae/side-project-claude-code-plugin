# Side Project Claude Settings

사이드 프로젝트를 혼자서 처음부터 끝까지 만들어보려고 하면, 생각보다 막막한 부분이 많습니다.

"어디서부터 시작하지?" "기획서는 어떻게 써?" "디자인은?" "개발은 어떤 순서로?"

이 플러그인을 설치하면, 그냥 "앱 만들고 싶은데"라고 말하는 것부터 첫 배포까지 한 흐름으로 이어집니다. 다음에 뭘 해야 하는지 고민할 필요가 없습니다.

---

## 설치

```bash
claude plugin install side-project-claude-settings --scope project
```

이것만 하면 됩니다. 필요한 동반 플러그인 4개는 첫 사용 시 자동으로 설치됩니다.

---

## 설치하면 일어나는 일

플러그인을 설치하는 순간, 프로젝트에 다음이 준비됩니다:

- 기획부터 이슈 생성까지 이어지는 **파이프라인 스킬 8개**
- 아이디어 검증, PM, 구현, 작업 관리를 위한 **동반 플러그인 4개** (자동 설치)
- 대화를 자동 기록하는 **hook**
- 문서가 바뀌면 알려주는 **변경 감지 hook**

이제 아이디어만 있으면 시작할 수 있습니다.

---

## 아이디어부터 첫 배포까지 — 전체 흐름

아래는 실제로 이 세팅을 쓰면 겪게 되는 흐름입니다. 각 단계에서 사람이 뭘 하고, Claude가 뭘 하는지 같이 적었습니다.

### 1. 아이디어 검증

> 사이드 프로젝트에서 가장 흔한 실수는 검증 없이 바로 만들기 시작하는 겁니다.

```
나: 운동 루틴 공유 앱 아이디어가 있는데, 해볼 만한지 검증해줘
```

**Claude가 하는 일:**
[slavingia/skills](https://github.com/slavingia/skills)의 `validate-idea`가 동작합니다. Gumroad 창업자가 만든 스킬로, Mom Test 방식으로 아이디어를 검증하고 Lean Canvas를 만들어줍니다.

타겟 고객을 좀 더 파고 싶으면, [phuryn/pm-skills](https://github.com/phuryn/pm-skills)의 제품 디스커버리 스킬이 고객 인터뷰 설계부터 우선순위 정리까지 해줍니다.

**사람이 하는 일:** 검증 결과를 보고 진행할지 판단합니다. 이 단계는 선택이라, 확신이 있으면 건너뛰어도 됩니다.

---

### 2. 기획

```
나: 앱 기획해줘
```

**Claude가 하는 일:**
세 명의 에이전트가 각각 다른 관점에서 토론합니다.
- "사용자가 진짜 원하는 건 뭐지?" (사용자 옹호자)
- "돈은 어떻게 벌지?" (비즈니스 전략가)
- "AI로 만들 수 있긴 한 거야?" (기술 현실주의자)

세 관점이 부딪히면서 균형 잡힌 결론이 나옵니다. 핵심 가치, MVP 범위, 유저 플로우를 정하고, 기획서(PRD)를 작성합니다. 유저 플로우 다이어그램과 화면 전환 흐름도도 HTML로 만들어줍니다.

**사람이 하는 일:** 기획서를 읽고, 방향이 맞는지 판단합니다. 수정할 부분이 있으면 말하면 됩니다.

**이 기획서가 이후 모든 단계의 출발점입니다.** 기획서 없이 디자인을 시작하려 하면 "먼저 기획서를 작성하세요"라고 안내합니다.

---

### 3. 디자인

```
나: 디자인 시스템 만들어줘
```

**Claude가 하는 일:**
기획서를 읽고, 앱 성격에 맞는 디자인 토큰(색상, 타이포, 간격)을 생성합니다. 금융 앱이면 블루, 음식 앱이면 오렌지 계열로. 그 토큰으로 버튼, 입력 필드, 카드, 네비게이션 같은 컴포넌트를 HTML로 만듭니다.

```
나: 화면 디자인해줘
```

기획서의 화면 정의를 읽고, 각 화면을 모바일 크기(390x844) HTML로 만듭니다. 브라우저에서 바로 열어볼 수 있습니다.

**사람이 하는 일:** 브라우저에서 디자인을 확인하고, 수정 요청을 합니다. "이 버튼 좀 더 크게", "색상 바꿔줘" 같은 식으로.

디자인 품질을 위해 `frontend-design` 스킬이 없으면 자동 설치됩니다. 실패하면 멈춥니다. 디자인 원칙 없이 만들면 "AI가 만든 티"가 나기 때문입니다.

---

### 4. 개발 계획

```
나: 개발 계획 세워줘
```

**Claude가 하는 일:**
기획서와 디자인을 분석해서 기술 아키텍처, 디렉토리 구조, 데이터 모델, API를 설계합니다.

기술 스택이 정해지면, 그 스택의 **베스트 프랙티스 스킬을 실시간 검색**해서 설치합니다. 6개월 전에 좋았던 스킬이 지금은 관리 안 될 수 있으니, 정적 목록이 아니라 매번 최신을 찾습니다. 이 단계는 건너뛸 수 없습니다.

**사람이 하는 일:** 아키텍처를 리뷰합니다. 기술 스택이나 구조에 의견이 있으면 반영합니다.

---

### 5. 배포 로드맵

```
나: 로드맵 만들어줘
```

**Claude가 하는 일:**
개발 계획을 마일스톤(M0~M3)으로 나누고, 각 작업을 분류합니다:
- **claude-task**: AI가 혼자 할 수 있는 것 (코딩, 테스트, 설정)
- **human-task**: 사람이 해야 하는 것 (외부 서비스 가입, 앱스토어 등록)

작업 순서를 자동으로 최적화합니다. 사람 작업에 의존하지 않는 claude-task를 먼저 배치해서, 사람이 Supabase를 세팅하는 동안 AI가 할 수 있는 건 미리 다 해놓습니다.

**사람이 하는 일:** 로드맵을 확인하고, 우선순위를 조정합니다.

---

### 6. 이슈 생성

```
나: 이슈 만들어줘
```

**Claude가 하는 일:**
로드맵을 읽고 GitHub Issues를 자동으로 만듭니다. 에픽 이슈 아래에 하위 작업이 연결되고, claude-task/human-task 라벨이 붙습니다. claude-task에는 어떤 AI가 봐도 동일하게 구현할 수 있을 만큼 명확한 스펙이, human-task에는 따라하기 쉬운 스텝바이스텝 가이드가 들어갑니다.

**사람이 하는 일:** 이슈 구조를 확인하고 승인합니다. 자동으로 만들되, 최종 판단은 사람이 합니다.

---

### 7. 구현

이제 이슈 보드에 할 일이 쌓여 있습니다. 여기서부터는 [superpowers](https://github.com/obra/superpowers)와 [garrytan/gstack](https://github.com/garrytan/gstack)이 주력입니다.

**Claude가 하는 일 (claude-task):**
- 코드를 바로 짜지 않고, 먼저 설계를 검토합니다 (brainstorming)
- 작업을 2~5분 단위로 쪼갭니다 (writing-plans)
- 서브에이전트를 배정해서 병렬로 개발합니다 (subagent-driven-development)
- 코드 리뷰를 하고, 테스트가 통과해야만 머지합니다 (TDD)
- 버그가 나면 4단계 근본 원인 분석으로 디버깅합니다

**사람이 하는 일 (human-task):**
- 이슈에 적힌 스텝바이스텝 가이드를 따라 외부 서비스를 세팅합니다
- Claude가 만든 PR을 리뷰하고 머지합니다
- 수동 QA로 최종 확인합니다

사람이 human-task를 처리하는 동안 Claude는 다음 claude-task를 미리 진행하고 있습니다. 서로 기다리는 시간을 최소화합니다.

---

### 8. 첫 배포

모든 마일스톤이 끝나면 develop에서 main으로 PR을 만들고, 빌드와 테스트를 통과하면 배포합니다.

---

## 배포 이후 — 유지보수

첫 배포에서 끝이 아닙니다. 운영하다 보면 버그도 나오고, 기능도 추가해야 합니다.

### 세션 관리

세션이 끊겨도 괜찮습니다. 모든 대화는 hook이 `docs/sessions/`에 자동 저장하고 있습니다. 다음 세션을 시작하면 superpowers가 이전 맥락을 읽고 이어서 작업합니다.

### 버그 수정

```
나: 로그인 화면에서 크래시가 나
```

superpowers의 `systematic-debugging`이 동작합니다. 증상 확인 → 가설 수립 → 근본 원인 분석 → 수정 → 검증, 4단계로 체계적으로 디버깅합니다. 수정 후 실제 동작이 확인되어야만 완료로 처리합니다.

### 기능 추가

새 기능이 필요하면, 기획서를 업데이트하고 같은 파이프라인을 다시 탑니다. PRD 수정 → 필요하면 디자인 추가 → 개발 계획 업데이트 → 로드맵에 이슈 추가 → 구현.

문서가 바뀌면 hook이 감지해서 "블루프린트 업데이트가 필요합니다"라고 알려줍니다. 문서가 흩어지는 걸 방지합니다.

### 프로젝트 현황 확인

언제든 "현황 보여줘"라고 하면 파이프라인 상태를 보여줍니다.

```
나: 현황 보여줘

[완료] 기획서 — docs/ssot/prd/2026-04-15-MyApp-기획서.md
[완료] 디자인 토큰 — docs/ssot/design/system/tokens.css
[완료] 화면 디자인 — 5개 화면
[완료] 개발 계획 — docs/ssot/dev/dev-plan.md
[진행] 배포 로드맵 — M1 진행 중 (12/20 완료)
[진행] GitHub Issues — 8개 완료, 12개 남음
```

모든 문서를 한 화면에서 보고 싶으면 `/product-blueprint`를 실행하면 기획, 디자인, 개발, 로드맵을 하나의 HTML로 통합해서 보여줍니다.

## 사람과 Claude의 역할 정리

| 단계 | 사람이 하는 것 | Claude가 하는 것 |
|------|-------------|----------------|
| 아이디어 검증 | 아이디어를 설명한다 | Mom Test, Lean Canvas로 검증한다 |
| 기획 | 방향을 판단하고 수정한다 | 3인 토론으로 PRD를 작성한다 |
| 디자인 | 브라우저에서 확인하고 피드백한다 | 토큰, 컴포넌트, 화면 HTML을 만든다 |
| 개발 계획 | 아키텍처를 리뷰한다 | 기술 설계 + 베스트 프랙티스 스킬을 설치한다 |
| 로드맵 | 우선순위를 조정한다 | 마일스톤 분류, 작업 순서 최적화를 한다 |
| 이슈 생성 | 구조를 승인한다 | GitHub Issues를 자동 생성한다 |
| 구현 | human-task 수행, PR 리뷰 | claude-task 구현, TDD, 코드 리뷰 |
| 유지보수 | 버그 보고, 기능 요청 | 체계적 디버깅, 파이프라인 재실행 |

핵심은 **사람은 판단하고, Claude는 실행한다**는 겁니다. 기획 방향, 디자인 피드백, 아키텍처 결정, 우선순위 — 이런 건 사람이 합니다. 문서 작성, 코드 생성, 이슈 관리, 디버깅 — 이런 건 Claude가 합니다.

---

## 순서가 알아서 잡히는 원리

각 스킬은 이전 스킬이 만든 파일이 있는지 확인합니다. 기획서가 없으면 디자인을 시작할 수 없고, 디자인 토큰이 없으면 화면을 만들 수 없습니다. 순서를 건너뛰려 하면 "먼저 이걸 하세요"라고 안내하고, 정말 건너뛰고 싶으면 `--skip` 옵션도 있지만 결과물에 경고가 남습니다.

디자인 품질을 위한 스킬 설치, 개발 스택 베스트 프랙티스 스킬 설치, 로드맵 작업 순서 최적화 — 이런 건 스킵 옵션 없이 강제됩니다.
