# GitHub Issues 기반 작업 관리

진행상황은 **GitHub Issues**로 관리한다. `gh` CLI를 사용한다.

## 구조

- **에픽 이슈**: 큰 단위의 마일스톤. `epic` 라벨 부여.
- **하위 작업 이슈**: 에픽 아래 구체적 작업. 에픽 이슈 링크를 본문에 포함.
- **라벨**:
  - `claude-task`: Claude Code가 독립적으로 수행 가능한 작업
  - `human-task`: 사람이 직접 수행해야 하는 작업 (외부 서비스 설정, 결제 등)
  - `epic`: 에픽 이슈 표시

## Claude 이슈 템플릿

```markdown
## 맥락
- 상위 에픽: #{에픽번호}
- 관련 문서: [링크]
- 전체 흐름 내 위치: {이전 작업} → **이 작업** → {다음 작업}

## 할일
- [ ] 구체적 작업 1
- [ ] 구체적 작업 2
- [ ] 구체적 작업 3

## 검증 방법
- [ ] 검증 항목 1
- [ ] 검증 항목 2
```

## Human 이슈 템플릿

```markdown
## 맥락
- 상위 에픽: #{에픽번호}
- 관련 문서: [링크]
- 전체 흐름 내 위치: {이전 작업} → **이 작업** → {다음 작업}

## 목적
{왜 이 작업이 필요한지}

## 단계별 가이드
- [ ] Step 1: {설명} → [서비스 링크](URL)
- [ ] Step 2: {설명}
- [ ] Step 3: {설명}

## 완료 기준
- [ ] {확인 항목}
```

## 이슈 생성 원칙

- 이슈 생성 전 사용자와 대화로 범위와 분류를 합의한다
- 이슈 하나는 하나의 명확한 작업만 담는다
- `claude-task` 이슈는 이슈 본문만 읽고 독립 수행 가능한 수준으로 작성한다
- `human-task` 이슈는 단계별 가이드가 처음 하는 사람도 따라할 수 있을만큼 자세해야 한다
- 인터넷 검색으로 최신 정보 기반 작성하고, 참고한 출처를 본문에 남긴다

## 이슈 설명 업데이트

- 사용자에게 기존 이슈 내용과 다른 수정 피드백을 받으면 해당 이슈의 설명(body)을 즉시 업데이트한다
- `gh issue edit {이슈번호} --body "..."` 명령으로 반영한다
- 이슈 설명은 항상 최신 합의 내용을 반영해야 한다
