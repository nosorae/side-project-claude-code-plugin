# Dev Plan (개발 계획서 생성)

기획서(PRD)와 디자인 결과물을 분석하여 `docs/ssot/dev-plan.md`를 생성하는 스킬입니다.

## 트리거 조건

다음과 같은 요청이 들어올 때 자동 발동:
- "개발 계획 세워줘", "dev plan 작성"
- "/dev-plan"
- "개발 시작하려는데 계획이 필요해"

## 전제조건

- `docs/ssot/` 디렉토리에 기획서(PRD) 파일이 존재해야 한다 (`docs/ssot/YYYY-MM-DD-*-기획서.md`)
- 기획서가 없으면 실행을 거부하고, `/app-plan` 스킬을 먼저 실행하도록 안내한다

## 실행 단계

### Step 1: 기획서 분석

**Actions:**

1. `docs/ssot/` 디렉토리에서 가장 최근 기획서 파일을 찾는다
2. 기획서에서 다음 정보를 추출한다:
   - 핵심 가치 제안
   - MVP 기능 목록
   - 기술 스택 (기획서에 명시된 경우)
   - 화면 목록 및 유저 플로우
3. 디자인 결과물이 있으면 함께 분석한다:
   - `docs/ssot/tokens.css` (디자인 토큰)
   - `docs/ssot/` (디자인 HTML)

4. **사용자 확인**: "기획서를 분석했습니다. 추가로 고려할 기술적 요구사항이 있나요?"

### Step 2: 기술 아키텍처 설계

기획서의 기술 스택을 기반으로 아키텍처를 설계한다.

**Actions:**

1. **프로젝트 구조 정의**
   - 디렉토리 구조 (트리 형태)
   - 주요 설정 파일 목록

2. **데이터 모델 설계**
   - 엔티티 정의 (필드, 타입, 관계)
   - DB 스키마 또는 Supabase/Firebase 구조

3. **API 설계** (필요한 경우)
   - 엔드포인트 목록
   - 요청/응답 형식
   - 인증 방식

4. **핵심 컴포넌트 정의**
   - 공유 컴포넌트 목록
   - 상태 관리 전략
   - 외부 라이브러리/서비스 의존성

5. **사용자 확인**: "기술 아키텍처 초안입니다. 수정할 부분이 있나요?"

### Step 3: 문서 생성 및 저장

**Actions:**

1. `docs/ssot/dev-plan.md` 파일 생성
2. 다음 구조로 작성:

```markdown
# [앱 이름] 개발 계획서

> **기획서**: [기획서 파일 경로]
> **작성일**: YYYY-MM-DD
> **기술 스택**: [스택 요약]

## 1. 기술 아키텍처

### 시스템 구성도
[텍스트 다이어그램]

### 기술 스택 상세
| 영역 | 기술 | 선택 이유 |
|------|------|----------|
| 프론트엔드 | ... | ... |
| 백엔드 | ... | ... |
| DB | ... | ... |
| 배포 | ... | ... |

## 2. 디렉토리 구조
[트리 형태]

## 3. 데이터 모델
[엔티티별 정의]

## 4. API 설계
[엔드포인트 목록]

## 5. 핵심 컴포넌트
[컴포넌트별 역할과 의존성]

## 6. 외부 서비스 연동
[서비스별 용도와 설정 방법]

## 다음 단계
- [ ] `/dev-roadmap`으로 배포 로드맵 생성
```

3. Git 커밋 + 푸시

**Expected Output:**
기술 아키텍처, 디렉토리 구조, 데이터 모델, API 설계, 핵심 컴포넌트가 포함된 `docs/ssot/dev-plan.md`

---

## 모델 선택 가이드

- 기획서 분석 + 아키텍처 설계: `opus` (복잡한 기술 판단)
- 문서 구조화 + 저장: `sonnet` (구조화된 문서 작성)

---

## 품질 체크리스트

- [ ] 기획서의 모든 MVP 기능이 아키텍처에 반영되었는가?
- [ ] 디렉토리 구조가 기술 스택에 적합한가?
- [ ] 데이터 모델이 유저 플로우의 모든 데이터를 커버하는가?
- [ ] 외부 서비스 의존성이 명시되었는가?
- [ ] 2주 MVP 범위에 맞는 복잡도인가? (과도한 설계 금지)
