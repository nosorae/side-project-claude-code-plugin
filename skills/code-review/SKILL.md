# Code Review (코드 리뷰)

커밋 또는 PR 전에 변경된 코드가 **개발 스택 베스트 프랙티스**와 **프로젝트 규칙**을 잘 지켰는지 크로스체크합니다.

> 4에이전트 병렬 리뷰 같은 무거운 방식이 아닌, 빠르고 실용적인 체크입니다.

## 실행 단계

### Step 1: 변경 사항 수집

**Actions:**

1. `git diff --cached` (staged) 또는 `git diff HEAD` (전체)로 변경된 코드를 수집한다
2. 변경된 파일 목록과 변경 라인 수를 파악한다

### Step 2: 체크 기준 수집

**Actions:**

1. **프로젝트 규칙 확인**:
   - `.claude/rules/` 디렉토리의 규칙 파일들을 읽는다
   - `CLAUDE.md` 파일이 있으면 읽는다
   - 프로젝트의 린트 설정 (`.eslintrc`, `ktlint`, `swiftlint` 등)이 있으면 참조

2. **개발 스택 베스트 프랙티스 확인**:
   - `docs/ssot/dev/dev-plan.md`에서 기술 스택과 설치된 베스트 프랙티스 스킬을 확인
   - 해당 스킬이 설치되어 있으면 그 스킬의 가이드라인을 기준으로 사용

### Step 3: 크로스체크

**Actions:**

변경된 코드를 아래 항목으로 체크한다:

1. **프로젝트 컨벤션 준수**
   - 네이밍 규칙 (파일명, 변수명, 함수명)
   - 디렉토리 구조 (dev-plan에 정의된 구조와 일치하는지)
   - import 순서, 코드 포맷팅

2. **스택 베스트 프랙티스**
   - 설치된 베스트 프랙티스 스킬의 권장 패턴을 따르는지
   - 안티패턴 사용 여부
   - 에러 처리 적절성

3. **기본 품질**
   - 하드코딩된 값 (매직 넘버, 하드코딩 URL 등)
   - 사용하지 않는 import/변수
   - TODO/FIXME 잔존 여부
   - 보안 이슈 (API 키 노출, SQL 인젝션 등)

### Step 4: 결과 보고

**Actions:**

```markdown
## 코드 리뷰 결과

**변경 파일**: N개 / **변경 라인**: +N / -N

### 이슈 (수정 필요)
- [ ] `파일명:라인` — 설명 (근거: 프로젝트규칙/베스트프랙티스)

### 제안 (선택)
- `파일명:라인` — 설명

### 통과
- 프로젝트 컨벤션: ✅
- 스택 베스트 프랙티스: ✅/⚠️
- 기본 품질: ✅
```

이슈가 0개면 "리뷰 통과 — 커밋/PR 진행하세요" 안내.

## 품질 체크리스트

- [ ] 변경된 모든 파일이 체크되었는가?
- [ ] 프로젝트 규칙과 베스트 프랙티스 기준이 명확한가?
- [ ] 오탐(false positive)이 최소화되었는가?
