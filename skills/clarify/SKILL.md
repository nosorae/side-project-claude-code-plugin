# Clarify (요구사항 구체화)

모호한 이슈나 요청을 사용자 인터뷰를 통해 구체적인 실행 가능 수준으로 만드는 스킬입니다.

## 트리거 조건

다음과 같은 요청이 들어올 때 자동 발동:
- "/clarify"
- "이슈 구체화해줘", "요구사항 정리해줘"
- "이슈들 정리 좀", "모호한 이슈 확인해줘"

## 실행 단계

### Step 1: 대상 수집

**Actions:**

1. **이슈 기반** (기본): `gh issue list --state open`으로 열린 이슈를 수집한다
2. **요청 기반**: 사용자가 특정 요청이나 아이디어를 제시한 경우 해당 내용을 대상으로 한다
3. 특정 이슈 번호가 지정되면 해당 이슈만 대상으로 한다

### Step 2: 모호함 분석

**Actions:**

각 대상의 본문을 읽고 다음 기준으로 구체화 필요 여부를 판단한다:

| 판단 기준 | 예시 |
|-----------|------|
| 본문이 비어있거나 짧음 | 제목만 있고 설명 없음 |
| 불확실성 마커 | "TBD", "미정", "기준필요", "추후 결정" |
| 범위 불명확 | "개선", "리팩토링" (구체적 대상 없음) |
| 기대 결과 불명확 | "더 좋게", "최적화" (측정 기준 없음) |
| 여러 해석 가능 | 한 문장이 2가지 이상으로 해석됨 |
| 전제 조건 누락 | 의존성이나 사전 작업 언급 없음 |

분석 결과를 사용자에게 요약 제시:
```
## 분석 결과

| 이슈 | 상태 | 구체화 필요 항목 |
|------|------|-----------------|
| #21 | ⚠️ 모호 | 브랜치 전략, 보호 규칙 범위 |
| #22 | ⚠️ 모호 | 리서치 플랫폼, 적용 위치 |
| #23 | ✅ 충분 | — |
```

### Step 3: 구조화된 인터뷰

**Actions:**

구체화가 필요한 항목에 대해 AskUserQuestion으로 질문한다.

**인터뷰 원칙:**
- **추론 우선**: 코드와 컨텍스트에서 답을 추론할 수 있으면 추론 결과를 제시하고 확인만 받는다
- **선택지 제공**: 열린 질문 대신 선택지 + 추천 옵션을 제시한다
- **한 번에 최대 4개 질문**: 질문이 많으면 라운드를 나눈다
- **결정 포인트만**: 구현 세부사항이 아닌 핵심 결정 사항만 질문한다

**질문 형식 예시:**
```
이슈 #{번호} "{제목}" 구체화:

1. [범위] A 방식 vs B 방식?
   → 추천: A (이유: ...)

2. [기준] 어떤 기준으로 적용?
   → 추천: X (이유: ...)
```

**라운드 진행:**
- 1라운드: 핵심 결정 사항 (범위, 방향)
- 2라운드 (필요 시): 세부 기준, 예외 처리
- 3라운드 (필요 시): 우선순위, 의존성

### Step 4: 이슈 업데이트

**Actions:**

구체화된 내용으로 이슈 본문을 업데이트한다:

```bash
gh issue edit {이슈번호} --body "..."
```

업데이트된 이슈 본문은 `rules/github-issues.md`의 템플릿(Claude/Human)을 따른다.

### Step 5: 결과 요약

**Actions:**

```markdown
## 구체화 완료

| 이슈 | 변경 전 | 변경 후 |
|------|---------|---------|
| #21 | "Git Flow 적용" (모호) | 브랜치 전략 + 보호 규칙 + init-project 연동 |
| #22 | "경쟁 리서치 추가" (모호) | Tier별 플랫폼 + app-plan Step 1 확장 |

## 다음 작업
- 위 이슈들을 순서대로 구현할 수 있습니다
- 추가 구체화가 필요한 이슈가 있으면 말씀해주세요
```
