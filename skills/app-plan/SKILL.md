# App Plan (앱 기획)

사용자가 제시한 앱 아이디어를 4단계에 걸쳐 체계적으로 검증하고 기획하는 스킬입니다.

## 트리거 조건

다음과 같은 요청이 들어올 때 자동 발동:
- "앱 기획해줘", "앱 기획 시작"
- "/app-plan", "app plan"
- "앱 만들고 싶은데", "앱 아이디어 있어"

## 실행 단계

각 단계가 완료되면 **사용자 확인을 받고** 다음 단계로 진행합니다.

### 단계별 문서화 규칙 (필수)
- **각 Step 완료 시마다** `docs/ssot/YYYY-MM-DD-[앱이름]-기획서.md` 파일을 업데이트
- Step 1 완료 → 기획서에 Step 1 결과 기록 → git commit + push
- Step 2 완료 → 기획서에 Step 2 결과 추가 → git commit + push
- Step 3, 4도 동일
- **세션이 끊겨도 기획서 파일을 읽으면 진행 상황 파악 가능**하게 작성
- 미완료 단계는 `(미완료)` 또는 `(Step N 대기중)` 표시

### Step 1: 아이디어 검증 (가장 중요) — 3인 에이전트 토론

앱 아이디어를 **3명의 독립 에이전트**가 각자 조사한 후 토론하여 검증합니다.

> **왜 토론인가?** 혼자 조사하면 확증 편향에 빠진다. 서로 다른 관점의 에이전트가 독립적으로 조사하고 반박하면 더 정확한 판단이 나온다.

**Actions:**

1. **아이디어 수집**
   - 사용자에게 앱 아이디어를 물어봅니다
   - 핵심 키워드 추출 (예: "운동 루틴 공유 앱")

2. **3개 에이전트 병렬 실행** (Task 도구로 동시 실행)

   각 에이전트는 WebSearch를 사용하여 **독립적으로 조사**합니다:

   **에이전트 A: 시장 옹호자 (Market Advocate)**
   - 역할: "이 아이디어가 성공할 수 있는 이유"를 찾는다
   - 조사 항목:
     - 시장 규모 (TAM/SAM/SOM)
     - 성장 트렌드 데이터
     - 유사 분야 성공 사례
     - 타겟 사용자 규모와 지불 의향
   - 검색 전략: "keyword market size 2026", "keyword app success story revenue"
   - **결론 형식**: "이 아이디어는 [근거]로 시장성이 있다/없다"

   **에이전트 B: 경쟁 분석가 (Competition Analyst)**
   - 역할: 경쟁 현황을 냉정하게 파악한다
   - 조사 항목:
     - 구글플레이/앱스토어 경쟁 앱 5개
     - 각 앱의 다운로드 수, 평점, 핵심 기능
     - 사용자 리뷰에서 추출한 불만/개선점
     - 경쟁사가 놓치고 있는 빈틈 (Gap)
   - 검색 전략: "keyword app 구글플레이", "keyword app reviews complaints"
   - **결론 형식**: "경쟁 상황은 [수준]이며, 진입 기회는 [근거]로 있다/없다"

   **에이전트 C: 악마의 변호인 (Devil's Advocate)**
   - 역할: "이 아이디어가 실패할 이유"를 적극적으로 찾는다
   - 조사 항목:
     - 이 분야에서 실패한 앱/서비스 사례
     - 진입 장벽과 리스크
     - "왜 이미 누군가 안 만들었는가?" 의문
     - 규제/법적 리스크
     - AI 코딩 기반으로 실행 가능한가? 기술적 난이도는?
   - 검색 전략: "keyword app failed why", "keyword startup challenge risk"
   - **결론 형식**: "이 아이디어의 가장 큰 리스크는 [1, 2, 3]이다"

3. **토론 및 종합 (opus)**

   3개 에이전트의 결과를 모아 **opus 모델이 최종 판정**:

   ```markdown
   ## 토론 결과

   ### 시장 옹호자의 주장
   [에이전트 A 결론 요약]

   ### 경쟁 분석가의 주장
   [에이전트 B 결론 요약]

   ### 악마의 변호인의 반박
   [에이전트 C 결론 요약]

   ### 최종 판정

   **진행 가치:** ⭐⭐⭐⭐☆ (5점 만점)

   | 항목 | 판정 | 근거 |
   |------|------|------|
   | 시장 규모 | ✅/⚠️/❌ | ... |
   | 경쟁 강도 | ✅/⚠️/❌ | ... |
   | 차별화 가능성 | ✅/⚠️/❌ | ... |
   | 실행 가능성 (AI 코딩 기반) | ✅/⚠️/❌ | ... |
   | 수익화 가능성 | ✅/⚠️/❌ | ... |

   **핵심 리스크:** ...
   **핵심 기회:** ...
   **최종 권고:** 진행 / 피봇 제안 / 중단
   ```

4. **사용자 확인**
   "3명의 에이전트가 토론한 결과입니다. 확인하시고 계속 진행할까요?"

**모델 선택:**
- 에이전트 A, B, C: `sonnet` (독립 웹 검색 + 분석, 병렬 실행)
- 토론 종합 + 최종 판정: `opus` (다각도 판단, 편향 보정)

**Expected Output:**
3개 관점의 독립 분석 + 토론 종합 + 별점 판정 + 진행/중단 권고

### Step 2: 핵심 가치 정의 — 3인 에이전트 토론

앱의 핵심을 **3명의 독립 에이전트가 각자 제안**한 후 토론하여 결정합니다.

> **왜 토론인가?** 가치 제안/기능/수익 모델은 관점에 따라 달라진다. 사용자, 비즈니스, 기술 관점이 충돌할 때 최적의 교집합을 찾아야 한다.

**Actions:**

1. **Step 1 결과를 입력으로 제공** (토론의 공통 근거)

2. **3개 에이전트 병렬 실행** (Task 도구로 동시 실행)

   **에이전트 D: 사용자 옹호자 (User Advocate)**
   - 역할: "사용자가 진짜 원하는 것"에 집중
   - 판단 항목:
     - 사용자 페인 포인트 우선순위
     - "이걸 안 쓰면 무슨 고통이 있는가?"
     - 가장 즉각적인 가치를 주는 기능 1개
     - 사용자 관점 가치 제안 문장
   - **결론 형식**:
     - 가치 제안: "[누가] [무슨 문제를] [어떻게 해결]"
     - 핵심 기능 1개 제안 + 이유
     - 사용자가 돈 낼 의향이 있는 기능

   **에이전트 E: 비즈니스 전략가 (Business Strategist)**
   - 역할: "어떻게 돈을 벌 것인가"에 집중
   - 판단 항목:
     - 수익 모델 추천 (근거 기반)
     - 경쟁 앱의 수익 모델 분석
     - 가격 책정 전략
     - 시장 포지셔닝 (프리미엄? 가성비? 니치?)
     - 사용자 획득 채널 (어디서 첫 100명?)
   - **결론 형식**:
     - 수익 모델: [모델명] + 가격대 + 근거
     - 포지셔닝: 한 문장
     - 첫 100명 확보 전략

   **에이전트 F: 기술 현실주의자 (Technical Realist)**
   - 역할: "AI 코딩으로 어디까지 가능한가, 기술적 리스크는 무엇인가"에 집중
   - 판단 항목:
     - 제안된 기능의 기술적 난이도 (상/중/하)
     - AI(클로드 코드)가 잘 생성할 수 있는 기능인가?
     - 기능별 구현 순서 제안 (의존성 기반)
     - 기술적 리스크 (외부 API 의존, 데이터 수집 등)
   - **결론 형식**:
     - 구현 가능성: ⭐⭐⭐⭐☆
     - 기능별 난이도 + AI 코딩 적합도
     - 기술적 리스크와 대응 방안

3. **토론 및 종합 (opus)**

   3개 에이전트의 제안을 모아 **opus 모델이 토론을 중재하고 최종 결정**:

   ```markdown
   ## 토론 결과

   ### 사용자 옹호자의 제안
   - 가치 제안: "..."
   - 핵심 기능: [기능명] — 이유: ...

   ### 비즈니스 전략가의 제안
   - 수익 모델: [모델명] — 가격: ... — 이유: ...
   - 포지셔닝: "..."

   ### 기술 현실주의자의 제안
   - 구현 가능성: ⭐⭐⭐⭐☆
   - 기능별 난이도 + AI 코딩 적합도
   - 기술적 리스크와 대응 방안

   ### 쟁점 및 합의

   | 쟁점 | 사용자 | 비즈니스 | 기술 | 최종 결정 |
   |------|--------|----------|------|----------|
   | 핵심 기능 | ... | ... | ... | **[결정]** |
   | 수익 모델 | ... | ... | ... | **[결정]** |
   | 제품 범위 | ... | ... | ... | **[결정]** |

   ### 최종 결정

   ## 핵심 가치 제안
   [한 문장 — 3인 토론으로 도출]

   ## 핵심 기능
   - **기능명**: 설명
   - **선정 이유**: 사용자 가치 + 수익화 가능 + AI 코딩 적합

   ## 부가 기능
   - 기능 1 — 우선순위: 상/중/하
   - 기능 2 — 우선순위: 상/중/하

   ## 수익 모델
   **선택:** [모델명]
   **가격:** ...
   **근거:** [비즈니스 전략가 분석 기반]

   ## 첫 100명 확보 전략
   1. ...
   2. ...
   ```

4. **사용자 확인**
   "3명의 에이전트가 토론한 결과입니다. 핵심 가치와 기능에 동의하시나요?"

**모델 선택:**
- 에이전트 D, E, F: `sonnet` (관점별 분석, 병렬 실행)
- 토론 중재 + 최종 결정: `opus` (다관점 종합, 최적 교집합 도출)

**Expected Output:**
3개 관점 토론 + 합의된 가치 제안 + 핵심 기능 1개 + 수익 모델 + 첫 100명 전략

### Step 3: 제품 범위 설정

전체 제품의 기능 범위를 정의하고 구현 우선순위를 결정합니다.

**Actions:**

1. **화면 리스트**

   선정 기준:
   - 핵심 기능 + 부가 기능에 필요한 모든 화면
   - 로그인/회원가입 간소화 (소셜 로그인 우선)
   - 온보딩은 최소화

   예시:
   ```markdown
   1. 스플래시/온보딩
   2. 메인 화면
   3. 핵심 기능 화면
   4. 부가 기능 화면들
   5. 설정 화면
   ```

2. **각 화면의 핵심 요소**

   화면별로:
   - 주요 UI 컴포넌트 (3~5개)
   - 필수 데이터
   - 사용자 액션

   예시:
   ```markdown
   ### 메인 화면
   - 리스트 (RecyclerView)
   - 추가 버튼 (FAB)
   - 필터/정렬 (선택적)
   **데이터:** 항목 리스트
   **액션:** 추가, 수정, 삭제
   ```

3. **기술 스택 결정 (AI First 원칙)**

   > **핵심 사고방식**: "내가 잘 아는 스택"이 아니라 **"AI(클로드 코드)가 가장 빠르고 정확하게 코딩할 수 있는 스택"**을 선택한다.

   **AI First 스택 선택 기준:**
   - AI 학습 데이터에 **예시가 풍부한** 기술 (커뮤니티 크고 문서 많은 것)
   - **보일러플레이트가 적은** 기술 (설정 코드 < 비즈니스 로직)
   - **단일 언어로 풀스택** 가능한 기술 (컨텍스트 스위칭 최소화)
   - AI가 **한 번에 동작하는 코드**를 생성하기 쉬운 기술

   **앱 유형별 AI 최적 스택 판단:**

   ```markdown
   ## 판단 기준: 앱의 핵심 특성

   Q1. 네이티브 기능(카메라, GPS, 센서 등)이 핵심인가?
       → YES: 안드로이드 네이티브 (Kotlin + Compose)
       → NO: Q2로

   Q2. 빠른 검증이 최우선인가?
       → YES: 웹앱/PWA (Next.js or Flutter Web)
       → NO: Q3로

   Q3. 구글플레이 배포가 필수인가?
       → YES + 웹도 필요: Flutter (단일 코드베이스)
       → YES + 앱만: KMP (Kotlin Multiplatform) or 네이티브
       → NO: 웹앱으로 충분
   ```

   **AI가 가장 잘 쓰는 스택 (2026년 기준):**

   ```markdown
   ### Tier 1: AI 코딩 최적 (추천)
   - **Next.js + TypeScript + Tailwind + Supabase**
     - AI 학습 데이터 최다, 예시 풍부
     - 프론트+백+DB 단일 프로젝트
     - Vercel 원클릭 배포
     - PWA로 모바일 대응 가능
     - 적합: SaaS, 웹앱, 대시보드, 도구형 앱

   - **Flutter + Dart + Firebase**
     - AI가 위젯 트리 생성 매우 잘함
     - 안드로이드+iOS+웹 동시 배포
     - 선언형 UI로 AI 코드 생성 정확도 높음
     - 적합: 모바일 우선 앱, 크로스 플랫폼

   ### Tier 2: AI 코딩 양호
   - **Kotlin + Jetpack Compose + Firebase**
     - Compose는 선언형이라 AI 친화적
     - 안드로이드 전용일 때 선택
     - 적합: 안드로이드 전용, 네이티브 기능 필수

   - **React Native + Expo + Supabase**
     - JavaScript 생태계 활용
     - Expo로 빌드/배포 간소화
     - 적합: 빠른 프로토타입, JS 익숙할 때

   ### Tier 3: 특수 목적
   - **Python + FastAPI + React**: AI/ML 모델 서빙 필요 시
   - **Cursor + Vibe Coding**: 프로토타입 극한 속도
   ```

   **백엔드/인프라 (AI 코딩 기준):**
   ```markdown
   ## BaaS (Backend as a Service) — MVP에 최적
   - **Supabase** (추천): PostgreSQL + Auth + Storage + Realtime
     - AI가 SQL 쿼리 생성 매우 정확
     - 무료 티어 넉넉
   - **Firebase**: NoSQL + Auth + Hosting + Functions
     - 실시간 기능 필요 시 유리
   - **Convex / Appwrite**: 신규 대안

   ## AI API
   - Claude API (Anthropic): 텍스트 분석/생성
   - OpenAI API: 범용 AI 기능
   - Google Gemini: 멀티모달
   ```

   **최종 스택 결정 시 사용자에게 근거와 함께 제안:**
   ```
   "이 앱은 [이유]로 [스택]이 AI 코딩에 가장 적합합니다.
   - AI가 잘 생성하는 이유: ...
   - MVP 속도 이점: ...
   - 배포 편의성: ..."
   ```

4. **우선순위 분류표**

   "없으면 안 되는 것" vs "v2로 미룰 것"

   ```markdown
   ## Must Have (MVP)
   - [ ] 핵심 기능 A
   - [ ] 데이터 저장/불러오기
   - [ ] 기본 UI/UX

   ## Nice to Have (v2)
   - [ ] 소셜 공유
   - [ ] 푸시 알림
   - [ ] 다크 모드
   - [ ] 통계/분석
   ```


5. **사용자 확인**
   "MVP 범위가 적절해 보이나요? 조정할 부분이 있나요?"

**모델 선택:**
- MVP 범위 설정: `sonnet` (기술적 판단 + 구조화)

**Expected Output:**
화면 리스트, 각 화면의 핵심 요소, 기술 스택, 우선순위 분류

---

### Step 4: 유저 플로우 정의

사용자가 앱을 열고 핵심 가치를 체험하기까지의 최단 경로를 정의합니다.

**Actions:**

1. **핵심 유저 플로우 1개 선정**

   질문:
   - "사용자가 가장 먼저 해야 할 일은?"
   - "핵심 가치를 가장 빠르게 전달하는 경로는?"

   예시:
   - "운동 루틴 앱" → 첫 루틴 생성 및 체크
   - "식재료 관리 앱" → 첫 식재료 등록 및 알림 설정

2. **플로우 다이어그램 (텍스트 기반)**

   ```markdown
   ## 핵심 유저 플로우: [플로우명]

   ```
   [앱 열기]
      ↓
   [스플래시 (1초)]
      ↓
   [메인 화면] ← 데이터 없음
      ↓ (사용자: FAB 클릭)
   [생성 화면]
      ↓ (사용자: 정보 입력)
   [확인 버튼 클릭]
      ↓ (시스템: DB 저장)
   [메인 화면] ← 데이터 표시
      ↓ (사용자: 항목 클릭)
   [상세 화면]
      ↓
   ✅ [핵심 가치 체험 완료]
   ```
   ```

3. **화면별 상세 액션**

   각 화면에서:
   - 사용자가 할 수 있는 액션
   - 시스템의 반응
   - 다음 화면으로 전환 조건

   ```markdown
   ### 1. 메인 화면
   **사용자 액션:**
   - FAB 클릭 → 생성 화면 이동
   - 항목 클릭 → 상세 화면 이동
   - 스와이프 → 삭제

   **시스템 반응:**
   - 데이터 로딩 (Room DB)
   - 리스트 표시
   - 빈 상태 처리 (Empty State)

   **예외 처리:**
   - 네트워크 오류 → 오프라인 모드
   - 로딩 실패 → 재시도 버튼
   ```

4. **화면 전환 흐름도**

   전체 화면 간 이동 경로:
   ```markdown
   ## 화면 전환 맵

   ```
   [메인]
      ├─→ [생성] → [메인] (완료 시)
      ├─→ [상세] → [수정] → [메인]
      └─→ [설정]
   ```
   ```

5. **3초 룰 체크**

   "앱 열고 3초 내에 사용자가 다음 액션을 알 수 있는가?"

   체크 포인트:
   - [ ] 메인 화면이 직관적인가?
   - [ ] 다음 액션이 명확한가? (FAB, 버튼 등)
   - [ ] 빈 상태에서도 가이드가 있는가?

5. **사용자 확인**
   "유저 플로우가 명확한가요? 개선할 부분이 있나요?"

**모델 선택:**
- 유저 플로우 정의: `sonnet` (구조화 + UX 판단)

**Expected Output:**
핵심 유저 플로우 1개, 화면별 상세 액션, 화면 전환 흐름도, 3초 룰 체크

## Step 5: 최종 기획서 저장

4단계 결과를 통합하여 마크다운 문서로 저장합니다.

**Actions:**

1. **파일 생성**
   - 위치: `docs/ssot/`
   - 파일명: `YYYY-MM-DD-[앱이름]-기획서.md`

2. **문서 구조**
   ```markdown
   # [앱 이름] 기획서

   > **한 줄 요약:** [핵심 가치 제안]
   > **목표:** 완성도 높은 MVP
   > **작성일:** YYYY-MM-DD

   ## 1. 아이디어 검증
   [Step 1 결과]

   ## 2. 핵심 가치
   [Step 2 결과]

   ## 3. MVP 범위
   [Step 3 결과]

   ## 4. 유저 플로우
   [Step 4 결과]

   ## 다음 단계
   - [ ] 프로젝트 세팅
   - [ ] 디자인 와이어프레임 (선택)
   - [ ] 개발 시작
   ```

3. **가독성 체크**
   - [ ] 각 섹션이 독립적으로 이해 가능한가?
   - [ ] 구체적 예시가 포함되었는가?
   - [ ] 5초 내에 핵심 내용 파악 가능한가?
   - [ ] 단계별로 명확히 분리되었는가?

4. **Git 커밋**
   ```
   Add: [앱이름] 기획서

   - 아이디어 검증 완료
   - MVP 범위 정의
   - 개발 범위 정의
   ```

5. **사용자에게 안내**


**모델 선택:**
- 문서 통합 및 저장: `sonnet` (구조화된 문서 작성)

## 모델 선택 가이드

### Opus 사용 (최고 품질 — 토론 종합 + 최종 판단)
- **Step 1 최종 판정**: 3인 에이전트(A,B,C) 토론 결과 종합, 편향 보정, 진행/중단 결정
- **Step 2 최종 결정**: 3인 에이전트(D,E,F) 제안 중재, 최적 교집합 도출

### Sonnet 사용 (균형 — 개별 에이전트 + 구조화)
- Step 1 에이전트 A, B, C: 독립 웹 검색 + 관점별 분석 (병렬 실행)
- Step 2 에이전트 D, E, F: 관점별 제안 생성 (병렬 실행)
- Step 3: MVP 범위 설정 (AI First 스택 판단 + 구조화)
- Step 4: 유저 플로우 정의 (구조화)
- Step 5: 최종 기획서 저장

### Haiku 사용 (빠르고 저렴)
- 검색 쿼리 생성
- 간단한 텍스트 파싱/분류

## 예시

### Example 1: 운동 루틴 앱

**User says:**
```
앱 기획해줘. 퇴근 후 운동하는 직장인들을 위한 루틴 공유 앱이야.
```

**Actions:**

1. **Step 1: 3인 에이전트 토론 (sonnet × 3 → opus 종합)**
   - 에이전트 A (시장 옹호자): "한국 피트니스 앱 시장 5,000억, 직장인 운동 트렌드 급증"
   - 에이전트 B (경쟁 분석가): "Home Workout 500만DL이지만 한국어 루틴 부족, 리뷰 불만: 커스텀 어려움"
   - 에이전트 C (악마의 변호인): "레드오션, 기존 앱 충성도 높음, 네트워크 효과 없음"
   - **opus 최종 판정:** ⭐⭐⭐⭐☆ — "직장인 특화 + 한국어 루틴으로 틈새 존재, 단 소셜 기능 제외하고 MVP"

   "3명의 에이전트가 토론한 결과입니다. 진행할까요?"

2. **Step 2: 3인 에이전트 토론 (sonnet × 3 → opus 종합)**
   - 에이전트 D (사용자 옹호자): "핵심 기능은 '오늘의 루틴 체크', 사용자는 복잡한 설정 싫어함"
   - 에이전트 E (비즈니스 전략가): "프리미엄 모델, 무료 3개 루틴 + 무제한 월 4,900원, 첫 100명: 에브리타임"
   - 에이전트 F (기술 현실주의자): "타이머는 v2, 체크리스트만이면 1주면 됨, AI 코딩 난이도 하"
   - **opus 최종 결정:** "체크리스트 기능만, 프리미엄 모델, 타이머/소셜은 v2"

   "핵심 가치에 동의하시나요?"

3. **Step 3 실행 (sonnet)**
   - 화면: 스플래시, 메인(루틴 리스트), 루틴 실행, 설정
   - AI First 스택 판단: 네이티브 기능 불필요 + 빠른 검증 우선 → **Next.js + Supabase (PWA)**
   - 근거: "AI가 React/Next.js 코드를 가장 정확하게 생성, PWA로 모바일 대응, Vercel 원클릭 배포"


   "MVP 범위가 적절한가요?"

4. **Step 4 실행 (sonnet)**
   - 핵심 플로우: 앱 열기 → 첫 루틴 생성 → 운동 시작 → 체크 완료
   - 3초 룰: FAB로 바로 루틴 추가 가능

   "유저 플로우가 명확한가요?"

5. **Step 5 실행 (sonnet)**
   - 파일 생성: `docs/ssot/2026-02-08-운동루틴앱-기획서.md`
   - Git 커밋 + 푸시

   "기획서가 저장되었습니다!"

**Result:**
체계적인 4단계 검증 + 실행 가능한 MVP 기획서

## 트러블슈팅

### 경쟁 앱 정보가 부족할 때
1. 검색어를 더 구체적으로 조정
2. 영어 키워드로 재검색
3. "Google Play Store [keyword] top apps 2026" 검색
4. 직접 구글플레이 링크 확인 요청

### 시장 규모 추정이 어려울 때
1. 유사 카테고리 시장 규모 참고
2. 타겟 사용자 수 × 예상 ARPU로 계산
3. "추정치" 명시 + 근거 설명
4. 보수적으로 계산

### MVP 범위가 너무 클 때
1. "없으면 안 되는 것"만 남기기
2. 사용자와 범위 재조정 논의
3. v2 Features로 과감히 미루기
4. "핵심 가치만 전달하는 최소 버전" 강조

### 유저 플로우가 너무 복잡할 때
1. "핵심 가치 체험"까지만 플로우 정의
2. 부가 기능은 별도 플로우로 분리
3. 최단 경로(Happy Path)만 먼저 정의
4. 예외 처리는 나중에
