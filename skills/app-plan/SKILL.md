# App Plan (앱 기획)

사용자가 제시한 앱 아이디어를 4단계에 걸쳐 체계적으로 기획하는 스킬입니다.

> **시장 조사/아이디어 검증은 별도 스킬**: 시장성 검증이 필요하면 먼저 `/market-research`를 실행하세요. 이 스킬은 순수 기획(PRD 작성)에 집중합니다.

> **도식화는 반드시 HTML/CSS로 생성한다. 아스키아트, 머메이드 금지.**

## 실행 단계

모든 단계를 **중간 확인 없이 연속 실행**합니다. 최종 결과를 한 번에 제시한 뒤 사용자 리뷰를 받습니다.

### 단계별 문서화 규칙 (필수)
- **각 Step 완료 시마다** `docs/ssot/prd/YYYY-MM-DD-[앱이름]-기획서.md` 파일을 업데이트
- 각 Step 완료 → 기획서에 결과 기록 → git commit + push
- **세션이 끊겨도 기획서 파일을 읽으면 진행 상황 파악 가능**하게 작성
- 미완료 단계는 `(미완료)` 또는 `(Step N 대기중)` 표시

---

### Step 1: 핵심 가치 정의 — 3인 에이전트 토론

앱의 핵심을 **3명의 독립 에이전트가 각자 제안**한 후 토론하여 결정합니다.

> **왜 토론인가?** 가치 제안/기능/수익 모델은 관점에 따라 달라진다. 사용자, 비즈니스, 기술 관점이 충돌할 때 최적의 교집합을 찾아야 한다.

**Actions:**

1. **사용자 아이디어 + 시장 조사 결과를 입력으로 제공** (`/market-research` 결과가 있으면 함께 참조)

2. **3개 에이전트 병렬 실행** (Task 도구로 동시 실행)

   **에이전트 D: 사용자 옹호자 (User Advocate)**
   - 역할: "사용자가 진짜 원하는 것"에 집중
   - 판단 항목:
     - 사용자 페인 포인트 우선순위
     - "이걸 안 쓰면 무슨 고통이 있는가?"
     - 가장 즉각적인 가치를 주는 기능 1개
     - 사용자 관점 가치 제안 문장
   - **결론 형식**:
     - 가치 제안: "[누가] [무슨 문제를] [어떻게 해결]"
     - 핵심 기능 1개 제안 + 이유
     - 사용자가 돈 낼 의향이 있는 기능

   **에이전트 E: 비즈니스 전략가 (Business Strategist)**
   - 역할: "어떻게 돈을 벌 것인가"에 집중
   - 판단 항목:
     - 수익 모델 추천 (근거 기반)
     - 경쟁 앱의 수익 모델 분석
     - 가격 책정 전략
     - 시장 포지셔닝 (프리미엄? 가성비? 니치?)
     - 사용자 획득 채널 (어디서 첫 100명?)
   - **결론 형식**:
     - 수익 모델: [모델명] + 가격대 + 근거
     - 포지셔닝: 한 문장
     - 첫 100명 확보 전략

   **에이전트 F: 기술 현실주의자 (Technical Realist)**
   - 역할: "AI 코딩으로 어디까지 가능한가, 기술적 리스크는 무엇인가"에 집중
   - 판단 항목:
     - 제안된 기능의 기술적 난이도 (상/중/하)
     - AI(클로드 코드)가 잘 생성할 수 있는 기능인가?
     - 기능별 구현 순서 제안 (의존성 기반)
     - 기술적 리스크 (외부 API 의존, 데이터 수집 등)
   - **결론 형식**:
     - 구현 가능성: ⭐⭐⭐⭐☆
     - 기능별 난이도 + AI 코딩 적합도
     - 기술적 리스크와 대응 방안

3. **토론 및 종합 (opus)**

   3개 에이전트의 제안을 모아 **opus 모델이 토론을 중재하고 최종 결정**:

   ```markdown
   ## 토론 결과

   ### 사용자 옹호자의 제안
   - 가치 제안: "..."
   - 핵심 기능: [기능명] — 이유: ...

   ### 비즈니스 전략가의 제안
   - 수익 모델: [모델명] — 가격: ... — 이유: ...
   - 포지셔닝: "..."

   ### 기술 현실주의자의 제안
   - 구현 가능성: ⭐⭐⭐⭐☆
   - 기능별 난이도 + AI 코딩 적합도
   - 기술적 리스크와 대응 방안

   ### 쟁점 및 합의

   | 쟁점 | 사용자 | 비즈니스 | 기술 | 최종 결정 |
   |------|--------|----------|------|----------|
   | 핵심 기능 | ... | ... | ... | **[결정]** |
   | 수익 모델 | ... | ... | ... | **[결정]** |
   | 제품 범위 | ... | ... | ... | **[결정]** |

   ### 최종 결정

   ## 핵심 가치 제안
   [한 문장 — 3인 토론으로 도출]

   ## 핵심 기능
   - **기능명**: 설명
   - **선정 이유**: 사용자 가치 + 수익화 가능 + AI 코딩 적합

   ## 부가 기능
   - 기능 1 — 우선순위: 상/중/하
   - 기능 2 — 우선순위: 상/중/하

   ## 수익 모델
   **선택:** [모델명]
   **가격:** ...
   **근거:** [비즈니스 전략가 분석 기반]

   ## 첫 100명 확보 전략
   1. ...
   2. ...
   ```

4. **Step 1 완료** — 기획서에 기록 후 Step 2로 즉시 진행

**모델 선택:**
- 에이전트 D, E, F: `sonnet` (관점별 분석, 병렬 실행)
- 토론 중재 + 최종 결정: `opus` (다관점 종합, 최적 교집합 도출)

**Expected Output:**
3개 관점 토론 + 합의된 가치 제안 + 핵심 기능 1개 + 수익 모델 + 첫 100명 전략

### Step 2: 제품 범위 설정

전체 제품의 기능 범위를 정의하고 구현 우선순위를 결정합니다.

**Actions:**

1. **화면 리스트**

   선정 기준:
   - 핵심 기능 + 부가 기능에 필요한 모든 화면
   - 로그인/회원가입 간소화 (소셜 로그인 우선)
   - 온보딩은 최소화

   예시:
   ```markdown
   1. 스플래시/온보딩
   2. 메인 화면
   3. 핵심 기능 화면
   4. 부가 기능 화면들
   5. 설정 화면
   ```

2. **각 화면의 핵심 요소**

   화면별로:
   - 주요 UI 컴포넌트 (3~5개)
   - 필수 데이터
   - 사용자 액션

   예시:
   ```markdown
   ### 메인 화면
   - 리스트 (RecyclerView)
   - 추가 버튼 (FAB)
   - 필터/정렬 (선택적)
   **데이터:** 항목 리스트
   **액션:** 추가, 수정, 삭제
   ```

3. **기술 스택 결정 (AI First 원칙)**

   > `references/tech-stack-guide.md`를 참조하여 기술 스택을 결정한다.
   > 핵심: "내가 잘 아는 스택"이 아니라 "AI가 가장 빠르고 정확하게 코딩할 수 있는 스택"을 선택한다.
   > 앱 유형별 판단 플로우 → Tier별 추천 스택 → BaaS 선택 → 사용자에게 근거와 함께 제안

4. **우선순위 분류표**

   "없으면 안 되는 것" vs "v2로 미룰 것"

   ```markdown
   ## Must Have (MVP)
   - [ ] 핵심 기능 A
   - [ ] 데이터 저장/불러오기
   - [ ] 기본 UI/UX

   ## Nice to Have (v2)
   - [ ] 소셜 공유
   - [ ] 푸시 알림
   - [ ] 다크 모드
   - [ ] 통계/분석
   ```


5. **Step 2 완료** — 기획서에 기록 후 Step 3으로 즉시 진행

**모델 선택:**
- MVP 범위 설정: `sonnet` (기술적 판단 + 구조화)

**Expected Output:**
화면 리스트, 각 화면의 핵심 요소, 기술 스택, 우선순위 분류

---

### Step 3: 유저 플로우 정의

사용자가 앱을 열고 핵심 가치를 체험하기까지의 최단 경로를 정의합니다.

**Actions:**

1. **핵심 유저 플로우 1개 선정**

   질문:
   - "사용자가 가장 먼저 해야 할 일은?"
   - "핵심 가치를 가장 빠르게 전달하는 경로는?"

   예시:
   - "운동 루틴 앱" → 첫 루틴 생성 및 체크
   - "식재료 관리 앱" → 첫 식재료 등록 및 알림 설정

2. **플로우 다이어그램 (HTML/CSS)**

   유저 플로우를 HTML/CSS로 시각화하여 `docs/ssot/prd/userflow.html` 파일로 저장한다.
   - 각 단계를 박스로 표현하고, 화살표로 흐름을 연결
   - 사용자 액션과 시스템 반응을 구분하여 색상 등으로 표시
   - 브라우저에서 바로 열어볼 수 있는 standalone HTML 파일로 작성

3. **화면별 상세 액션**

   각 화면에서:
   - 사용자가 할 수 있는 액션
   - 시스템의 반응
   - 다음 화면으로 전환 조건

   ```markdown
   ### 1. 메인 화면
   **사용자 액션:**
   - FAB 클릭 → 생성 화면 이동
   - 항목 클릭 → 상세 화면 이동
   - 스와이프 → 삭제

   **시스템 반응:**
   - 데이터 로딩 (Room DB)
   - 리스트 표시
   - 빈 상태 처리 (Empty State)

   **예외 처리:**
   - 네트워크 오류 → 오프라인 모드
   - 로딩 실패 → 재시도 버튼
   ```

4. **화면 전환 흐름도 (HTML/CSS)**

   전체 화면 간 이동 경로를 HTML/CSS로 시각화하여 `docs/ssot/prd/screen-map.html` 파일로 저장한다.
   - 각 화면을 노드로, 전환을 화살표로 표현
   - 전환 조건(사용자 액션)을 화살표 위에 레이블로 표시
   - 브라우저에서 바로 열어볼 수 있는 standalone HTML 파일로 작성

5. **3초 룰 체크**

   "앱 열고 3초 내에 사용자가 다음 액션을 알 수 있는가?"

   체크 포인트:
   - [ ] 메인 화면이 직관적인가?
   - [ ] 다음 액션이 명확한가? (FAB, 버튼 등)
   - [ ] 빈 상태에서도 가이드가 있는가?

**모델 선택:**
- 유저 플로우 정의: `sonnet` (구조화 + UX 판단)

**Expected Output:**
핵심 유저 플로우 1개 (`docs/ssot/prd/userflow.html`), 화면별 상세 액션, 화면 전환 흐름도 (`docs/ssot/prd/screen-map.html`), 3초 룰 체크

## Step 4: 최종 기획서 저장

4단계 결과를 통합하여 마크다운 문서로 저장합니다.

**Actions:**

1. **파일 생성**
   - 위치: `docs/ssot/prd/`
   - 파일명: `YYYY-MM-DD-[앱이름]-기획서.md`

2. **문서 구조**
   ```markdown
   # [앱 이름] 기획서

   > **한 줄 요약:** [핵심 가치 제안]
   > **목표:** 완성도 높은 MVP
   > **작성일:** YYYY-MM-DD

   ## 1. 핵심 가치
   [Step 1 결과]

   ## 2. MVP 범위
   [Step 2 결과]

   ## 3. 유저 플로우
   [Step 3 결과]

   ## 시장 조사 (별도)
   > `/market-research`로 작성된 시장조사 문서가 있으면 여기에 링크: `docs/ssot/prd/YYYY-MM-DD-[앱이름]-시장조사.md`

   ## 다음 단계
   - [ ] 프로젝트 세팅
   - [ ] 디자인 와이어프레임 (선택)
   - [ ] 개발 시작
   ```

3. **가독성 체크**
   - [ ] 각 섹션이 독립적으로 이해 가능한가?
   - [ ] 구체적 예시가 포함되었는가?
   - [ ] 5초 내에 핵심 내용 파악 가능한가?
   - [ ] 단계별로 명확히 분리되었는가?

4. **Git 커밋**
   ```
   Add: [앱이름] 기획서

   - 아이디어 검증 완료
   - MVP 범위 정의
   - 개발 범위 정의
   ```

#### product-blueprint 자동 업데이트

product-blueprint.html의 기획 탭을 자동 업데이트한다. `/product-blueprint --tab=기획` 실행.

5. **사용자에게 최종 결과 제시 및 리뷰 요청**
   ```
   기획서가 저장되었습니다!

   위치: docs/ssot/prd/YYYY-MM-DD-[앱이름]-기획서.md

   ## 전체 기획 결과 요약
   - Step 1 (핵심 가치): [가치 제안 + 수익 모델 요약]
   - Step 2 (MVP 범위): [화면 수 + 기술 스택 요약]
   - Step 3 (유저 플로우): [핵심 플로우 요약]

   수정할 부분이 있으면 말씀해주세요.
   특정 Step만 다시 진행하거나, 세부 내용을 조정할 수 있습니다.

   ## 다음 추천
   기획서가 확정되면 `/design-system-to-figma`로 디자인 시스템을 바로 이어서 생성할 수 있습니다.
   기획과 디자인을 동시에 진행하려면 지금 바로 요청하세요.
   ```

**모델 선택:**
- 문서 통합 및 저장: `sonnet` (구조화된 문서 작성)
